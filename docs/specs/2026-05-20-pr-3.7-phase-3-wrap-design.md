# PR 3.7 — Phase 3 wrap: happy-path admin E2E + `v0.3.0` tag

> Design spec for the PR that closes Phase 3 of the Al Hewal build.
> Written 2026-05-20 after deciding to drop the previously-deferred
> bilingual PDF export from scope (the CSV export shipped in PR #47

## Context

Phase 3 of the master plan delivers the Admin Command Center: auth,
admin shell, properties CRUD, image upload + reorder + hero pick,
admin management with invite flow, and the leads journal. The
catalog of merged PRs (#14 → #48) covers every master-plan
deliverable in Phase 3 **except** the closure work:

- A happy-path E2E that exercises the whole admin → public-site
  flow end-to-end.
- A `v0.3.0 — Admin Command Center` git tag marking the phase
  complete.
- A documented carve-out for the bilingual PDF export the master
  plan originally listed as Phase 3 scope.

This PR ships those three items as a single change so Phase 4
opens against a cleanly-closed Phase 3.

## Goals

1. Verify that an admin can sign in, create a property, upload an
   image, publish it, and have it appear on the public site —
   end-to-end against a real preview deploy.
2. Document the PDF-export decision in the canonical plan + handoff
   docs so the next session does not re-open it.
3. Cut the `v0.3.0` tag against the post-merge `main` commit.

## Non-goals

- Implementing the bilingual PDF export. Explicitly cut.
- Backfilling additional E2E specs for routes already covered by
  `tests/e2e/admin-*.spec.ts`. The 11 existing admin specs plus

- Wiring the new spec into CI's main run. CI does not (and will
  not) carry the live `BLOB_READ_WRITE_TOKEN` or production
  Supabase URL; the spec stays preview-only.

## Architecture

### The happy-path spec

One new file: `tests/e2e/admin-happy-path.spec.ts`.

One `test()` block: one full create+upload+publish flow through
the EN admin UI, then bilingual public-side verification on both
`/en/properties/<slug>` and `/ar/properties/<slug>`. The admin form's
own AR rendering is already covered by `admin-property-form.spec.ts`
(the spec's "AR /admin/properties/new renders RTL with Arabic
chrome" case); the value-add here is verifying that an
admin-created property surfaces correctly in both public locales,
which is the actual bilingual coverage the master plan asks for.

```ts
// pseudo-shape
import { test, expect } from '@playwright/test';
import { loginAsAdmin } from './helpers/admin-auth';

const isPreview = Boolean(process.env.BLOB_READ_WRITE_TOKEN);

test.describe('admin happy path', () => {
  test.skip(!isPreview, 'preview-only — requires real Blob + Supabase');

  for (const locale of ['ar', 'en'] as const) {
    test(`super_admin can create, upload, publish, then see on /${locale}`, async ({
      context,
      page,
    }) => {
      const propertyName = `E2E Happy Path ${Date.now()}`;
      await loginAsAdmin(context, { tier: 'super_admin' });

      // 1. Create
      await page.goto(`/${locale}/admin/properties/new`);
      await fillPropertyForm(page, { name: propertyName, locale });
      await page.getByRole('button', { name: /save|حفظ/i }).click();
      await page.waitForURL(/\/admin\/properties\/[^/]+\/edit/);
      const propertyId = page.url().match(/properties\/([^/]+)\/edit/)![1];

      // 2. Upload
      await page.setInputFiles('input[type="file"]', './tests/e2e/fixtures/property-test.png');
      await expect(page.getByTestId('property-image-card').first()).toBeVisible({
        timeout: 15_000,
      });

      // 3. Publish
      await page.getByRole('button', { name: /publish|نشر/i }).click();
      await expect(page.getByText(/published|منشور/i)).toBeVisible();

      // 4. Public-side check
      const slug = await page.getByTestId('property-slug').innerText();
      await page.goto(`/${locale}/properties/${slug}`);
      await expect(page.getByRole('heading', { level: 1 })).toContainText(propertyName);
      await expect(page.locator('img[alt][src*="vercel-storage"]').first()).toBeVisible();

      // 5. afterEach handles cleanup
      cleanupQueue.push(propertyId);
    });
  }

  test.afterEach(async ({ context }) => {
    for (const id of cleanupQueue.splice(0)) {
      // Use the admin context.request so the cookie set by loginAsAdmin propagates
      // (per the established memory: do NOT use the standalone request fixture).
      await context.request.delete(`/api/properties/${id}`);
    }
  });
});
```

The above is sketch only — the plan-writing step turns this into
final code.

### Test fixture

Reuses the existing `tests/fixtures/sample-400x300.jpg` (already
used by `tests/unit/lib/image-pipeline.test.ts`). No new binary is
added. The upload pipeline (`/api/upload`) resizes + converts to
AVIF/WebP, so the source format does not matter; the test asserts
that the resized variant appears in `property-images-grid`.

### Env gate

`test.skip(!process.env.BLOB_READ_WRITE_TOKEN, …)` reuses the same
signal `admin-upload.spec.ts` already keys off. The whole reason
this spec is preview-only is that the upload step needs a real
Blob token — the absence of that token is the truest gate.

CI does not set this env var (see `.github/workflows/ci.yml`); the
spec will skip silently in regular CI runs. Preview deploys that
have the token will execute it.

### Cleanup

`test.afterEach` archives + deletes each property created during
the test, mirroring the pattern used by `admin-management.spec.ts`
when it creates throwaway admin records. Cleanup uses
`context.request.delete(...)` so the admin session cookie set by
`loginAsAdmin` propagates (per the
`feedback_playwright_context_request` memory: the standalone
`request` fixture does not carry context cookies).

If cleanup throws, the test still passes — we do not want a flaky
DB write to mask a green happy-path. The error is logged via
`test.info().annotations`.

## Data flow

```
Playwright test process
  │
  ▼ injects signed admin cookie via loginAsAdmin
Next.js dev/preview server
  │
  ├──── POST /<locale>/admin/properties/new (form submit)
  │       └─► Server Action → Supabase INSERT properties (draft)
  ▼
  ├──── POST /api/upload (multipart from input[type=file])
  │       └─► sharp resize/AVIF + Blob put + Supabase INSERT property_images
  ▼
  ├──── POST publish action
  │       └─► Supabase UPDATE properties.status = 'published'
  ▼
  └──── GET /<locale>/properties/<slug>
          └─► public reader sees published row, hero image renders
  │
  ▼ afterEach
  └──── DELETE /api/properties/<id> (soft-delete + image cleanup)
```

## Error handling

| Failure                                | Behavior                                                                                                                                                                              |
| -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `BLOB_READ_WRITE_TOKEN` not set        | `test.skip` — silent, no failure. CI runs hit this path.                                                                                                                              |
| Form submit returns validation error   | Test fails on the `waitForURL` timeout. Surface the form's error banner text in the failure message via `await page.getByRole('alert').textContent()`.                                |
| Upload fails (network, Blob rejection) | Test fails on the `getByTestId('property-image-card')` visibility timeout. The `admin-upload.spec.ts` 503-detection pattern remains in place as a second-layer assertion for CI runs. |
| Publish fails                          | Test fails on the published-state badge assertion.                                                                                                                                    |
| Public-side page returns 404           | Test fails on the heading assertion. Indicates either revalidation lag or RLS leak — both are real bugs we want to surface, not flake.                                                |
| `afterEach` cleanup DELETE fails       | Logged as a test annotation. Does NOT fail the test. Manual sweep in the preview DB is acceptable.                                                                                    |

## Testing approach

This PR IS the test. There is no production code to unit-test.

Verification of the spec itself:

1. Run against the local dev server with a real Blob token exported
   (`BLOB_READ_WRITE_TOKEN=...`); confirm both locales pass.
2. Run against the preview deploy at `al-hewal.vercel.app` with
   `PLAYWRIGHT_BASE_URL=https://al-hewal.vercel.app`; confirm both
   locales pass.
3. Run against CI's default env (no Blob token); confirm both
   specs skip cleanly.

## Free-tier impact

Zero runtime impact. No new endpoints, deps, or tables. The spec
runs only against environments that already have the resources
provisioned.

Preview-deploy Blob writes during test runs are negligible — the
fixture is < 20 KB and each run cleans up after itself.
