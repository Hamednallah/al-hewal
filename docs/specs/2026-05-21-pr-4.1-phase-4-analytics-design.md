# PR 4-A — Phase 4 (first half): Analytics + My Profile

> Design spec for the first PR of Phase 4. Per the durable user
> preference [[feedback_phase_as_one_pr]], a phase ships as 1 PR
> (2 if too big). Phase 4 totals ~1550-1850 LOC, which is over
> the 1-PR ceiling, so it splits in two:
>
> - **PR-A (this PR):** Analytics dashboard + My Profile page.
> - **PR-B (next PR):** TOTP enrolment + enforcement for
>   super_admin.
>
> The split line is the cleanest natural feature boundary:
> user-facing data + profile is one concern; security hardening
> with its own design surface (recovery codes, QR generation,
> library choice) is another.
>
> Two scope amendments captured during brainstorming on
> 2026-05-21:
>
> - The originally-planned SVG KSA heatmap is replaced by a
>   top-10 cities bar chart from `leads.city` aggregation.
> - The `page_views_daily` rollup + cron migrations are deferred
>   (production volume does not yet justify MV infrastructure).

## Context

Phase 4 of the master plan delivers admin-users-and-analytics.
Admin Management already shipped in Phase 3 (PR #33 + #39/#40
hardening), so what remains in Phase 4 is:

1. **Strategic Analytics dashboard** (this PR).
2. **My Profile page** (change email/password, last-login surface
   — placeholder exists at `/admin/profile`).
3. **TOTP enrolment for super_admin**.

The user chose "full analytics in one PR" during the Phase 4
brainstorm — KPI cards + Recharts line + bar + geographic view,
all together rather than incrementally. This spec is that PR.

## Goals

1. **Analytics:** Replace the `/admin/analytics` placeholder
   with a working dashboard reading from production data. Give
   the Al Hewal owner / sales lead a glanceable view of funnel
   performance (leads + WhatsApp clicks) AND inventory status
   (published properties, top property, top cities).
2. **My Profile:** Replace the `/admin/profile` placeholder
   with a real page showing the signed-in admin's email, tier,
   status, and last-login, and exposing two actions: change
   email and change password. Both round-trip Supabase Auth.
3. Stay free-tier-safe — all reads are server-side, indexed,
   bounded, and live inside the admin route group which is
   `force-dynamic` anyway. Zero client-side data fetching for
   the dashboards.

## Non-goals

- **No `pg_cron` rollup migration.** The dashboard does not depend
  on `page_views_daily`. Indexed COUNT queries over `leads` +
  `whatsapp_clicks` remain O(log n) and stay fast at any volume
  Al Hewal will see before the first big marketing push. If/when
  raw `page_views` is needed (page-views-per-property, geo
  rollups, etc.), introduce MVs in a follow-up PR.
- **No SVG KSA heatmap.** The brainstorm replaced the master
  plan's "bespoke SVG KSA heatmap from `leads.region` aggregation"
  with "top 10 cities bar chart from `leads.city` aggregation."
  Saves ~150 LOC + a 50 KB SVG asset + the maintenance cost of
  keeping admin-boundary path data current. Cities is also more
  actionable for the sales team than regions.
- **No TOTP enrolment.** That ships in PR-B (the second half of
  Phase 4), with its own design surface around recovery codes,
  QR generation, and enforce-on-login policy.
- **No client-side filtering on analytics.** Dashboards are
  read-only; filters can ship in a follow-up if asked for.
- **No avatar / display-name fields on My Profile.** The admin
  identity is the email; everything else is read-only metadata.
  Display-name + avatar can ship later if the team grows.

## Data shape

Four reader functions in `src/lib/data/admin-analytics.ts`.
Each is purely a Supabase read; none take parameters today
(date range is hard-coded to `last 30 days` to keep the surface
small). Future PRs can lift that.

```ts
// All four assume the caller has already established admin auth
// (the page.tsx awaits requireAdmin()).

interface KpiSnapshot {
  leadsLast30Days: number;
  whatsappClicksLast30Days: number;
  publishedProperties: number;
  topPropertyByLeads: { id: string; titleAr: string; titleEn: string; leadCount: number } | null;
}

interface LeadsPerDayPoint {
  date: string; // ISO yyyy-mm-dd
  leadCount: number;
}

interface LeadsBySourcePoint {
  source: 'whatsapp' | 'contact_form' | 'call_click';
  leadCount: number;
}

interface LeadsByCityPoint {
  city: string;
  leadCount: number;
}

export async function getKpiSnapshot(): Promise<KpiSnapshot>;
export async function getLeadsPerDay(): Promise<LeadsPerDayPoint[]>; // 30 points, zero-filled
export async function getLeadsBySource(): Promise<LeadsBySourcePoint[]>;
export async function getTopCities(): Promise<LeadsByCityPoint[]>; // top 10, descending
```

### Zero-fill semantics for the line chart

`getLeadsPerDay` returns exactly 30 points (today through 29 days
ago). Days with zero leads are still in the result with
`leadCount: 0`, so the line chart renders a continuous x-axis
instead of jumping over missing days. The aggregation happens in
SQL via `generate_series` joined left against
`select date_trunc('day', created_at) as d, count(*) from leads group by d`.

## Components

All under `src/components/admin/`.

| Component                | Purpose                                                                     | Notes                                                                                                                                 |
| ------------------------ | --------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| `AnalyticsKpiCards.tsx`  | Server component; renders 4 cards in a 1×4 (md) / 2×2 (sm) / 1×4 (xs) grid  | Reuses the existing `SummaryCard` shape from `/admin/page.tsx` for visual consistency — same Forest Teal value + Brass label pattern. |
| `AnalyticsLineChart.tsx` | Client component; Recharts `<LineChart>` of 30-day leads/day                | Hydrates with server-passed data; no client-side fetch. Empty-state message inside the chart area when all values are zero.           |
| `AnalyticsBarChart.tsx`  | Client component; Recharts `<BarChart>`. Reused for both source + city bars | Props: `data`, `valueKey`, `labelKey`, `emptyMessage`.                                                                                |

The line + bar charts are client components only because Recharts
requires browser-side measurement of the container for responsive
sizing. The page itself is a server component that does the data
reads, then passes the resulting arrays as props.

## Recharts dependency

`recharts@^2.13` — the version that supports React 19. Adds
~90 KB gzipped to client bundles on pages that import it.
**Acceptable** because:

- It only loads on `/admin/analytics`. The public site never
  imports Recharts.
- The admin route is `force-dynamic` and not optimised for first
  paint the way the public catalog is.
- Recharts has no transitive deps with concerning licenses
  (MIT throughout).

Added via `pnpm add recharts@^2.13` per CLAUDE.md (exact version,
PR body documents the rationale).

## Error handling

| Failure                      | Behavior                                                                                                                                                                                                  |
| ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Supabase query throws        | The page-level RSC catches at the outer `try/catch` (added) and renders an `AdminPlaceholder` with `body="Analytics is temporarily unavailable."` — same pattern as other admin pages. No partial render. |
| One reader returns zero rows | Charts show the "No data yet" empty-state message in their plot area. KPI cards show `0` / `—`. Page does NOT render an error.                                                                            |
| Reader times out (>3s)       | Page-level try/catch handles like the throw case. Future PR can add `Suspense` + streaming if this becomes common at scale.                                                                               |

## Free-tier impact

- **Recharts:** +90 KB gzipped on `/admin/analytics` only.
- **Function invocations:** one per analytics page load. Bounded
  by admin behavior — at most a few dozen requests per day.
- **DB reads:** four queries per page load. All hit indexed
  columns (`leads.created_at`, `leads.property_id`,
  `leads.source`, `leads.city`, `whatsapp_clicks.created_at`,
  `properties.status`). At Al Hewal's volume (≤ 1k rows
  cumulatively for months) each query is sub-millisecond. At
  1M rows the COUNT(\*) over leads with a date filter is still
  fast given the existing `leads_created_at_idx`.
- **No new tables, no migrations, no env vars.**
